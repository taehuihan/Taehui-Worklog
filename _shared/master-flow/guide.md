### 1) 뎁스(= 플로우 단계 / flow step) = 프레임 네이밍의 `/` 깊이
> ⭐ 뎁스는 "화면 전환"이 아니라 **「플로우 단계」** 다 (`/` 한 칸 = 다음 단계). 그래서 화면을 안 바꾸는 **전진 모달**도 한 단계로 센다(아래). 밴드 라벨 "Depth N"·상수 `DEPTH_GAP`은 표기 그대로 유지 — 개념만 flow step. (2026-07-09 워딩 개정)
- `vai/onboarding`=D1 · `vai/tutorial`=D2 · `vai/home`=D3 · `vai/home/*`=D4
- `vai/home/` 아래(대화·입력·추천질문·이용약관·구독·출처·복사·북마크·신고·신고접수·검색·블룸버그)는 **전부 같은 뎁스(D4)**
- **같은 화면 내 동작·오버레이(정보 모달·토스트·부모 복귀만 하는 확인 모달)** = **같은 단계** → 같은 뎁스·**별도 행에 나열**, 커넥터 없음. 이름은 하이픈(`홈-코인모으는법`) or 같은 이름. 예) 블룸버그·chat 모달들은 그 화면과 같은 뎁스.
- **다른 화면으로 전진하는 「전환 모달」(모달 안 버튼 → 다른 결과 화면)** = **플로우 단계** → **슬래시로 한 단계 깊게** 두고 **부모→모달→결과 커넥터 연결**(§5·§9). 예) 칭찬저금통 `선물선택`, 학습커스텀 초기화 확인 모달. → **하이픈=같은-화면 모달 / 슬래시=전진 모달** 로 네이밍 구분. (2026-07-09 신설)
- ⚠️ **슬래시 없는 이름 = 뎁스로 올리지 않는다.** 같은 화면이 **시간·조건으로 형태만 자동 전환**되는 것(예: 홈 전체노출형→추천형, "무료체험 5일째")도 "같은 화면 내"라 새 뎁스 ❌ → §2-1 **상태(같은 행)** 로 둔다. 진짜 뎁스는 슬래시(**플로우 단계 진행** = 유저의 페이지 이동 또는 전진 모달)로만 올라간다. (리아 학습커스텀 추천형 D2 오배치 교정, 2026-07-08 — 슬래시 없는 홈 형태변화를 플로우 진행으로 착각)

### 2) 행 배치
- **chat 본체·상태(대화·입력·추천질문) = 같은 행** (같은 화면의 상태)
- **모달은 같은 `/chat`이라도 별도 행**으로 분리
- 한 뎁스 안에서 여러 행 가능, **한 행 화면 수 제한 없음** (밴드가 콘텐츠 폭으로 늘어남)
- **분기 행 순서**: 이전 화면에서 **트리거 컴포넌트 위치 순서대로 위→아래** 배치 — 화면 위쪽 트리거 → 위쪽 분기 행, 아래쪽 트리거 → 아래쪽 분기 행 (SSOT 동결 전 이관, 2026-07-06)

### 2-1) 레이블 모델 (기준섹션 16037 실측 — ⚠️ 의미 혼동 주의)
- **서브레이블 = 뎁스(열)별 콘텐츠 그룹명**. 화면이 있는 **각 뎁스 열 시작 x**에, 행이 여러 개면 행마다(y 다르게) 단다. 예) D1 "인텔리전스 홈" / D2 "설명"·"상세" / D3 "대화". **「유저 세그먼트(무료 유저/유료 유저)」나 「행 그룹명」으로 쓰지 말 것** — D1에만 붙는 오류 발생(블룸버그, 2026-06-26).
- **상태레이블 = 화면별 상태명**(무료/유료/유료 CTA/태블릿 등), 각 화면 x에 나란히. **단일 화면(상태 안 나뉨)은 상태레이블 생략**(기준섹션 설명·대화엔 없음).
- **같은 화면의 다른 상태(무료/유료) = 같은 행 나란히** + 상태레이블로 구분. **다른 화면 = 다른 행**(분기행, §5). 둘은 별개 축: 상태는 가로(상태레이블), 화면 차이는 세로(행).

### 2-2) 유저 세그먼트(구매 상태) 규칙 (2026-06-26 신설 → 2026-06-29 재교정)
유저가 **미구매(무료) vs 구매·구독(유료)** 세그먼트에 따라 보는 화면/플로우가 다를 수 있다.
- **⭐ 핵심: 세그먼트가 다른 진입점·다른 플로우를 의미하면 → 별도 행 그룹으로 분리한다.** 무료 유저 행 그룹 / 유료 유저 행 그룹이 세로로 완전히 분리됨. (모범 답안: `16524:28640`)
- **세그먼트에 따라 플로우가 갈리면** → 각 세그먼트의 **진입(첫) 화면 서브레이블 뒤에 "(무료 유저)" / "(유료 유저)"** 표기. 진입에서 한 번 명시하면 이후 화면은 같은 세그먼트로 간주(반복 표기 ❌). 예) 홈무료 서브 "인텔리전스 홈 (무료 유저)" / 홈유료 서브 "인텔리전스 홈 (유료 유저)".
- **같은 화면, 콘텐츠만 다름**(같은 UI에 값/콘텐츠만 무료↔유료) → 기존 §2-1 **상태레이블**(무료/유료), 같은 행 나란히.
- ⚠️ **§5는 세그먼트 내에서도 항상 적용** — 같은 세그먼트 안에서도 *다른 트리거→다른 화면*이면 **분기행으로 분리**한다. (예: 홈무료의 chevron→설명, 카드→상세는 같은 무료 유저지만 트리거가 달라 **다른 행**.)

> ⚠️ 교정 이력: 2026-06-29 1차 교정 "세그먼트는 배치 축이 아니라 라벨" → **틀렸음**. 모범 섹션(`16524:28640`) 실측으로 재교정. 세그먼트는 라벨이기도 하고 행 그룹 배치 기준이기도 하다. [[feedback_rule_in_worksteps]]

### 5) 커넥터 규칙 (갱신)
- **커넥터는 플로우 단계 간 이동만 표기.** 같은 단계(같은 행·다른 행)끼리는 **연결 안 함** — **같은-화면 모달·상태·토스트는 나열만**. (단 **전진 모달**은 §1대로 슬래시로 한 단계 깊으므로 부모→모달→결과 커넥터 연결)
- **조건 레이블(분기 조건 텍스트·다이아몬드) 표기 안 함**
- **출발점 = 고정 모델 (내정보 방식, 항상 이것만 사용)**: 트리거 컴포넌트 bbox를 읽어 **그 모서리(X·Y 모두)** 에서 출발. 화면 1개든 팬아웃이든 동일. **금지**: 팬아웃에서 "프레임 모서리에 정렬, Y만 트리거에서" 같은 편의 — X도 반드시 컴포넌트 기준. 내부 트리거는 컴포넌트 모서리에서 출발 → 최근접 프레임 모서리로 빠진 뒤 라우팅(Y=컴포넌트 center)
- **도착점 = 도착 화면 좌변의 세로 중앙** (다중 도착은 중앙 기준 ±20px 분산). 출발=컴포넌트 모서리 ↔ 도착=화면 좌변 중앙 (대칭). 화살표 `ARROW_EQUILATERAL`
- **다중 상태 화면 출발 (조건부)**: 트리거→도착이 상태 불문 같음 → **맨 우측 화면 1개** / 상태별 도착이 다름(다중출발) → **각 상태 화면 버튼에서 각각** (구버전 "무조건 맨우측"은 폐기. 무료/무료체험 사례)
- 화면 통과 금지 (OVER 우회 / midX 분리). **같은 행 형제 화면도 화면** — 트리거가 행 왼쪽이면 형제들 위(레이블 밴드)로 우회 (정기구독중→해지 사례)
- **첫 방향전환 = 화면 우측 +20px 이후** (`firstBend_x ≥ 트리거화면.right + 20`): 출발 후 트리거 화면을 **벗어나서** 꺾는다. 화면 안/가장자리(+20 전)에서 꺾지 않음. (구버전 +10 삭제했다가 **+20로 부활** — 화면 이탈을 깔끔히. `connect()` 빌더가 강제·검증)

### 6) 트리거 인식 (커넥터가 화면 통째에 붙어 있을 때)
- **트리거는 화면이 될 수 없다** → 무조건 가장 가까운 아이콘/버튼
- 커넥터의 **실제 bbox 시작점**(화면에 닿는 끝점)을 구해, 그 점을 **포함(거리 0)** 하거나 **최근접**하는 인터랙티브 컴포넌트 = 트리거
- **magnet(AUTO) 값으로 추정 금지** — 실제 렌더된 bbox를 봐야 정확 (AUTO는 타겟 방향 가장자리로 빠져 빗나감)
- 후보 필터: INSTANCE/COMPONENT·버튼·아이콘·**스타일된 텍스트 링크** (배경·본문 텍스트만 제외 — 링크 텍스트는 트리거 가능)
- 전제: 커넥터를 트리거 근처에서 시작해 그려야 정확
- **트리거 선별**: `findOne` 첫매치 금지 — **전체 구문 매칭**(부분매칭 ✗) + INSTANCE/링크 우선으로 정확히 1개 특정 ("무료 체험 관리하기"를 "관리하기"로 오인한 사례)
- **메뉴행 트리거 = 행 우측 아이콘**: 라벨 텍스트 모서리에서 출발 ❌ → 라벨로 행을 **식별**만 하고, 출발은 **그 행 우측의 아이콘/요소 bbox 우측**(셰브론 등). 소스 커넥터가 출발한 위치의 요소를 그대로 잡는 것과 동일. 독립 카드/링크는 그 자신 우측. (언어/알림이 텍스트 x867이 아니라 아이콘 x1150에서 출발해야 하는 사례)
- **규칙1.5 버튼조상 (2026-07-06)**: 텍스트 라벨 매치 시 **좁은 버튼형 조상**(INSTANCE/COMPONENT, h≤80, 폭<0.5W)이 있으면 **그 버튼 우측 모서리**에서 출발 — CTA·pill 버튼용. 넓은 메뉴행(≥0.5W)은 건너뛰어 우측끝아이콘 규칙 보존. 배경: 비율 상수(0.55W/0.3W/0.5W)가 태블릿 1280에서 CTA(397px)를 전부 탈락시켜 텍스트 폴백 출발·행 우측 book 아이콘 오인 발생 (RA 학습 커스텀, `realacademy/master-flow/2026-07-06.md`)

### 트리거 맵 추출 = 위·아래 양방향 (누락 0 설계)
커넥터만 따라가면(위→아래) *커넥터 자체가 빠진 전환*(예: 챌린지 안내가 소스에 없던 때)을 영영 놓침. **버튼 전수(아래→위)** 가 안전망.
1. **소스 커넥터 전수** 읽기 — 모든 홈 상태·일부만 ❌·position 기반도 포함
2. **출발점 → 리프 트리거** — 출발이 컨테이너(카드)면 **도착 라벨로 안쪽 버튼 특정** (card→결과확인 ⇒ "결과 확인" 버튼, card→종목선택 ⇒ "수정하기" 버튼)
3. **버튼 전수 교차검증** (`crossCheckTriggers`, connect-builder.js):
   - **커버**(자기/조상(카드)/자손이 커넥터 출발) → 자동 ✓ (검토 X)
   - **제외**(per-flow 제외목록: 탭바·토글·조건부모달 등) → 자동 (검토 X)
   - **⚠️검토**(나머지) → 사람이 **의심 소수만** "연결/제외" 1회 판정. 제외는 목록에 기억 → 다음엔 자동
   - 부담 = "전수 검토" ❌ → "의심 소수 1회 판정" ✅. 전수 열거하되 자동 분류로 검토량 최소화.

### 8) 라우팅 — 화면 겹침·몰림 방지
- **화면은 섬, 커넥터는 갭(바닷길)로만.** 출발·도착만 화면에 닿고 중간 꺾임은 갭 안에서
- **수직** = 화면 사이 갭 안쪽 레인(가장자리에 붙이지 말 것) · **수평 우회** = 화면 top보다 위(레이블 밴드)
- **1→다 팬아웃**: 갭 폭 활용해 리저 간격 벌리고(예 100px) **도착 행 순서로 레인 배정** → 교차 최소화. 좁으면 `DEPTH_GAP` 확대
- **레인 전구간 클리어런스**: 세로 레인 X는 **하강하며 지나는 모든 행**에서 화면과 안 겹쳐야 함. 출발 높이의 가로 여백만 보면 아래 행 화면을 관통함. 다른 행을 통과하는 깊은 하강은 **밴드 안쪽 ✗ → 인터밴드 갭(밴드 끝~다음 밴드 시작)으로** 라우팅. (사례: 챌린지 내상금내역 레인 x1600이 D1 모달 행 결과모달 관통 → x3150 갭으로 이동)

### 9) 확인 모달 (Basic dialog: 확인/취소) — 전진 여부로 2종 (2026-07-09 §1 반영)
- **비전진(부모 복귀) 확인 모달** = 확인해도 부모로 돌아가거나 닫힘, **새 결과 화면 없음** → 같은-화면 오버레이 → 그 화면 뎁스의 **모달 행에 나열**, **부모↔모달 커넥터 없음**(어디서 떴는지는 트리거 위치로).
- **전진 확인 모달** = 확인 → **다른 결과 화면**으로 이동 → **전환 모달(§1)** → 슬래시로 **한 단계 깊게**, **부모→모달→결과 커넥터 모두 연결**(출발 = 각 버튼). "취소"(부모 복귀)는 표시 안 함. 예) 학습커스텀 초기화 확인 모달(커스텀→모달→추천형설정완료).

### 10) 기존 섹션 업데이트 워크플로 (2026-06-29 신설) — 신규 생성과 별개
이미 Master에 만든 섹션에 **소스 변경(교체·추가·삭제)을 반영**하는 작업. "신규 생성"과 워크플로가 다르다.

**0. 소스 특정** — Master 섹션명 ↔ 소스 페이지/섹션명 대응으로 추측. **첫 매칭만 "이 소스(ID) 맞죠?" 1회 확인** (한 페이지에 섹션 여럿·이름 미묘차 대비).

**1. 화면 식별 — ⚠️⚠️ 가장 자주 깨지는 곳**
- **네이밍에 하드코딩 금지.** `인텔리전스홈`·`/` 같은 이름 필터로 화면을 잡으면 **다른 네이밍의 새 화면을 통째로 놓친다** (2026-06-29 `튜토리얼1·2` 누락 사고).
- 소스 섹션의 **모바일 화면 후보를 전수**로 잡아라: 폭 ~375(300~440) 프레임/인스턴스. **제외**: 컴포넌트 라이브러리(보통 별도 y 영역에 모임)·variant 정의(`plan=`, `Property=`, `=` 포함)·화면 내부 요소(`CTASection`, `Frame 117…` 등 내부 컴포넌트).
- **diff 결과(추가/삭제/변경 목록)는 실행 전 사람이 시각 확인** — 수치/이름만으로 화면 누락·split을 못 잡는다.

**2. diff = 이름 매칭** — 동일 이름=교체, 소스에만=추가, Master에만=삭제. (Master 클론은 소스 이름 유지, 노드 ID는 추적 불가 → 이름이 키)

**3. 삭제 → Archive 섹션** — `🗄️ Archive — 소스 삭제 화면`(Master 가장 우측 끝, 없으면 생성)으로 **이동**(본 섹션에서 제거) + 화면마다 **출처 섹션·감지 날짜** 라벨. 본 섹션의 관련 커넥터·레이블도 정리. ("지운 게 아니라 모아둠" → 복구·검토 가능)

**4. 추가 → 가이드대로 배치** — 뎁스(§1), 행(§2·§2-2 세그먼트), 커넥터(§5). 진입점은 **소스 네이티브 커넥터 실측**(추정 ❌, §6). ⚠️ 새 화면이 **뎁스 구조를 바꾸면**(예: 튜토리얼이 홈 앞 진입 → 튜토리얼 D1, 홈 이하 +1 시프트) 기존 화면 전체를 재배치.

**5. 교체 → 같은 위치에 새 클론** — 높이(h)가 다르면 §4 수직공식으로 **그 행 아래 전체 cascade 재배치 + 커넥터 재계산**. (높이 같으면 위치만)

**6. rename = 삭제+추가로 처리** — 이름 매칭이라 소스에서 이름만 바꾸면 옛 이름→Archive, 새 이름→추가. (실제 같은 화면이라도. split도 동일: `설명`→`설명1`·`설명2`)

**7. 섹션 크기 재조정 + 우측 섹션 cascade** — 화면 추가/뎁스 시프트로 섹션 width·height가 커지면 **bg·섹션 크기 갱신** + 그 **우측의 모든 섹션을 `right+500` 기준으로 차례로 밀어** 겹침 제거(§6 섹션 위치 규칙=직전 섹션 우측+500 유지). ⚠️ 신규 생성과 달리 업데이트는 섹션이 **커지면서 옆 섹션·Archive를 침범**한다(2026-06-29 블룸버그 4750→6000이 Archive 침범). 크기 바꾼 뒤 **전체 섹션 겹침 검사** 필수.

**8. 검증** — 9단계 수치(행간격·도착center·관통0) + `verifyRowBranching` + 섹션 겹침0 + **시각 대조**(화면 누락·split·뎁스시프트 확인).

> ⚠️ 교훈(2026-06-29 블룸버그 업데이트): **워크플로 자체의 버그가 가장 위험**. 화면 식별 필터가 네이밍 하드코딩이라 튜토리얼을 놓쳤다. "기억 리셋 후 재현"처럼, **diff 로직이 틀리면 빠진 화면은 영영 안 보인다** → 화면 전수 + 시각 확인이 안전망. [[feedback_rule_in_worksteps]]

### 작업 순서 가드 (SSOT 16097:3930 §6 체크리스트에 박음)
- **0단계 트리거 맵**: 소스 화면별 인터랙티브 요소(bbox)→도착 매핑(네이티브 커넥터 endpoint 우선) → 뎁스·커넥터 출발점 판단의 입력. **빌드 전에** 수행
- **0단계 → 뎁스 판정 (슬래시 없는 화면 오배치 방지, §1)**: 각 전환을 **유저 내비게이션(페이지 이동 → 뎁스+1) vs 자동·조건 상태전환(시간조건 등 → 같은 뎁스 상태, §2-1)** 으로 분류. **슬래시 없는 화면을 새 뎁스에 두려 하면 멈추고 재확인** — 같은 화면의 형태변화(전체노출형↔추천형 등)를 플로우 진행으로 착각한 사례. (리아 학습커스텀, 2026-07-08)
- **0단계 → 행 배치 도출 (「다른 화면=분기행」 누락 방지)**: 트리거맵에서 **한 출발화면이 *다른 트리거*로 *다른 도착화면* N개로 분기하면 → 분기행 N개**. 같은 화면의 다른 스테이트(조건·값만 차이)만 같은 행. ⚠️ 「설명/상세(무료)」를 "무료 유저 경로"라는 편의로 한 행에 묶은 게 누락 사례 — 둘은 홈무료의 다른 트리거(chevron vs 카드)로 가는 **다른 화면**이라 분기행. (블룸버그 v2, 2026-06-26)
- **4·5·6단계 §2 포인터**: 화면/라벨 y는 §2 수직공식으로 계산(즉흥 상수 금지). 공식은 §2에만, 체크리스트엔 링크만(SSOT 유지)
- **9단계 수치 자기검증**: 각 행 `screen_y == 이전행 bottom + 837` · `DEPTH_GAP == 800` · `도착 y == 도착화면 center` 코드 assert (**시각검증 前**)

## 행 폭 한계 규칙 (2026-06-19, v2 시절) — ⚠️ 구버전 (현행 모델로 대체됨)

> **❗ 이 규칙은 v3에서 폐기.** v3는 네이밍=뎁스 모델이라 "한 행 최대 3화면" 제한이 없고
> (모달/액션 7화면이 한 행), 커넥터도 뎁스 간만 존재한다. 아래는 v2 시절 기록.

> **배경**: v2 섹션(16210:11148)에서 D4 검색 화면 5개를 한 행에 나열했더니
> 폭이 D5 컬럼을 침범했다. 표준 밴드폭(770)이 콘텐츠를 못 덮어
> "헤더는 짧은데 화면은 옆 Depth로 흘러가는" 오류가 발생.
> 아래 3규칙은 이 충돌을 **사전 차단**하기 위한 것.

### ① 한 Depth row 최대 화면 수 = 3개

화면 간 수평 step = **395px** (SW 375 + 내부 gap 20). ← 실측값. (구문서의 SG=40·step415는 오류)
n개 화면의 행 폭 = `395 × (n−1) + 375 = 395n − 20`.
검증: 2화면=770(=표준 밴드폭), 5화면=1955(=band-d4 실제폭). ✓

```
395n − 20 ≤ DEPTH_GAP(1270)  →  n ≤ 3.27
→ 한 row당 최대 3화면. 4개 이상이면 반드시 ③로 처리.
```

### ② 밴드(헤더) 폭 = 콘텐츠 기준 동적 산출

```
밴드 폭 = max(해당 Depth 가장 넓은 row의 화면 폭, 770)
       = max(395·n_max − 20, 770)
```

- **고정 770 금지.** 밴드를 콘텐츠 폭으로 그리면 침범이 **밴드끼리 겹쳐 즉시 보임** → 밴드 자체가 오류 감지기.
- 예) D4 검색 5개 → 밴드 폭 = `395×5 − 20 = 1955`.

### ③ 행 폭 초과(4화면+) 처리 정책

한 row가 3화면을 넘으면 둘 중 하나로 처리하고 **반드시 가이드/로그에 명시**:

- **(A) row 분할** — 3개씩 줄바꿈해 세로로 쌓기. 레이아웃 깔끔, **기본 추천**.
- **(B) 열 확장** — 해당 Depth 폭을 넓히고 **이후 모든 Depth를 우측 shift**.
  한 줄로 꼭 보여줘야 할 때만.
  ```
  shift = (초과 Depth의 최대 row 폭) − 770
  → 이후 Depth 전체(밴드·서브/상태레이블·화면·도착 커넥터)를 +shift
  ```
  - 예) D4 검색 5개: `1955 − 770 = +1185` → D5 전체 우측 이동.
  - 커넥터: **출발(고정) vertex만 남기고 도착측 vertex 전부 +shift** 하면 형태 보존.
  - 새 갭 = (넓힌 밴드 끝) → (다음 밴드 시작) = 표준 500 유지.

### 검증 패스 (배치 후 필수)

각 화면 `[x, x+SW]` 가 **다음 Depth 밴드 시작 x** 를 넘는지 자동 체크.
넘으면 침범 경고 → ② 밴드 폭 재산출 + ③ 처리. (현재 누락돼 있던 단계)

## 핵심 코드 패턴

### 커넥터 생성 함수 (mkVec)

```javascript
function mkVec(pts, name, sec) {
  var xs = pts.map(function(p){return p[0];});
  var ys = pts.map(function(p){return p[1];});
  var ox = Math.min.apply(null, xs), oy = Math.min.apply(null, ys);
  var v = figma.createVector();
  v.vectorNetwork = {
    vertices: pts.map(function(p){return {x: p[0]-ox, y: p[1]-oy};}),
    segments: pts.slice(0,-1).map(function(_,i){return {start:i, end:i+1};}),
    regions: []
  };
  v.x = ox; v.y = oy;
  v.fills = [];
  v.strokes = [{type:'SOLID', color:{r:0.478,g:0.478,b:0.478}}];
  v.strokeWeight = 2;
  v.strokeCap = 'ARROW_EQUILATERAL'; // 도착 vertex에만 적용
  v.name = 'conn:' + name;
  sec.appendChild(v);
  return v;
}
```


## 막힌 지점 (Blockers)

### 1. Figma Section 내 노드 접근 불가

**현상**: `figma.getNodeById('sectionsNodeId')` → null 반환  
**원인**: Figma Plugin API에서 비현재 페이지의 SectionNode는 children이 빈 배열로 반환됨  
**해결**: `await figma.setCurrentPageAsync(targetPage)` 로 페이지 먼저 전환 후 접근

```javascript
// 잘못된 방법
const sec = figma.getNodeById('16073:3930'); // null

// 올바른 방법
await figma.setCurrentPageAsync(masterPage);
const sec = figma.getNodeById('16073:3930'); // 정상 접근
```

### 2. createConnector() 미지원

**현상**: `figma.createConnector()` → 함수 없음 오류  
**원인**: Figma Plugin API에 커넥터 생성 함수가 없음  
**해결**: `figma.createVector()` + `vectorNetwork` 수동 설정으로 커넥터와 동일한 벡터 생성

### 3. vectorNetwork 누적 오프셋 버그

**현상**: 기존 커넥터를 업데이트할 때 좌표가 이상하게 이동  
**원인**: `vectorNetwork` 재설정 시 기존 `v.x`, `v.y` 오프셋에 새 좌표가 누적됨  
**해결**: 업데이트 전 반드시 `v.x = 0; v.y = 0;` 리셋 필요

### 4. Cross-page 화면 복제 좌표 혼동

**현상**: Task 페이지에서 clone한 화면을 Master 섹션에 appendした 후 좌표가 맞지 않음  
**원인**: `appendChild` 전후로 좌표 기준이 page-absolute → section-relative로 바뀜  
**해결**: `sec.appendChild(clone)` 직후에 섹션 기준 상대 좌표로 x/y 설정

### 5. 커넥터 겹침 문제

**현상**: 동일 Depth 갭을 통과하는 커넥터가 겹쳐서 어떤 연결인지 불명확  
**해결**: 갭을 N등분해 midX 분리
- 커넥터 2개: 갭 내 1/3 지점, 2/3 지점
- 커넥터 3개: 1/4, 1/2, 3/4 지점
- 타겟이 먼 커넥터 → midX를 갭 좌측에 배치
